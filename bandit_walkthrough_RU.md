> Сайт: [overthewire.org/wargames/bandit](https://overthewire.org/wargames/bandit/)

## Что такое Bandit?

Bandit — это wargame (учебная CTF-игра) для абсолютных новичков в Linux. Каждый уровень — это отдельный пользователь на удалённом сервере (`bandit0`, `bandit1`, ..., `bandit33`). Задача каждого уровня — найти пароль для входа под следующим пользователем.

Игра постепенно учит работе с Linux: файловой системой, правами доступа, сетевыми утилитами, архивами и Git'ом. Прохождение занимает несколько часов, но даёт отличную базу.

### Как подключаться

```bash
ssh banditN@bandit.labs.overthewire.org -p 2220
```

Где `N` — номер уровня (0, 1, 2...). Порт всегда `2220`.

## Level 0 — Первый вход

**Задача:** Подключиться к серверу по SSH.

```bash
ssh bandit0@bandit.labs.overthewire.org -p 2220
# Пароль: bandit0
```

**Что такое SSH?**  
SSH (Secure Shell) — протокол для безопасного удалённого управления сервером. Всё соединение зашифровано. Флаг `-p 2220` указывает нестандартный порт (стандартный — 22).

После входа нас встречает приветственный баннер OverTheWire. Мы внутри!

## Level 0 → 1

**Задача:** Пароль хранится в файле `readme` в домашней директории.

```bash
ls           # смотрим, что есть в текущей директории
cat readme   # читаем содержимое файла
```

**Объяснение:**
- `ls` — показывает список файлов (от *list*)
- `cat` — выводит содержимое файла в терминал (от *concatenate*)
- При входе по SSH мы автоматически попадаем в домашнюю директорию (`~`)

Пароль виден в выводе `cat`. Записываем его и переходим дальше.

## Level 1 → 2

**Задача:** Пароль в файле с именем `-`.

Запустил `ls` — вижу файл `-`. Набрал `cat -` — терминал завис. Оказывается, минус означает стандартный ввод (stdin), и `cat` начал ждать ввода с клавиатуры.

```bash
cat ./-
# или
cat < -
```

**Объяснение:**
- `./` явно указывает путь к файлу, не позволяя оболочке трактовать `-` как флаг
- `< -` тоже работает — перенаправляет содержимое файла в stdin
- Урок: файлы с нестандартными именами требуют явного указания пути через `./`

## Level 2 → 3

**Задача:** Пароль в файле `spaces in this filename`.

Пробую `cat spaces in this filename` — bash считает каждое слово отдельным аргументом и ругается, что файлов `spaces`, `in`, `this`, `filename` не существует.

```bash
cat "spaces in this filename"
# или
cat spaces\ in\ this\ filename
# или просто начать вводить имя и нажать Tab!
```

**Объяснение:**
- Пробел — разделитель аргументов в bash
- Кавычки `"..."` объединяют слова в один аргумент
- Обратный слэш `\` экранирует специальный символ
- **Лайфхак:** Tab-автодополнение само добавит `\` перед пробелами

## Level 3 → 4

```bash
cd inhere
ls        # пусто!
```

**Объяснение:**

Скрытые файлы — не настоящая защита, а лишь соглашение. Обычно это конфиги (`.bashrc`, `.gitconfig` и т.д.).

## Level 5 → 6

**Задача:** Файл с паролем в `inhere` (20 поддиректорий!) со свойствами: человекочитаемый, 1033 байта, не исполняемый.

```bash
cd inhere
```

**Объяснение:**
- `find` — поиск файлов по условиям

`find` — одна из самых мощных утилит Linux. Стоит хорошо её изучить.

## Level 6 → 7

**Задача:** Файл с паролем находится **где-то на сервере** со свойствами: владелец `bandit7`, группа `bandit6`, 33 байта.

```bash
find / -user bandit7 -group bandit6 -size 33c 2>/dev/null
cat /var/lib/dpkg/info/bandit7.password
```

**Объяснение:**
- Начинаем поиск с `/` — корня всей файловой системы

- `-group bandit6` — группа файла `bandit6`
- `2>/dev/null` — перенаправляем stderr (поток ошибок, дескриптор `2`) в `/dev/null` — «мусорное ведро» Linux. Без этого вывод завален сотнями строк "Permission denied"

## Level 8 → 9

**Задача:** Пароль — строка в `data.txt`, встречающаяся ровно один раз. Все остальные строки повторяются много раз.

```bash
sort data.txt | uniq -u
```

**Объяснение:**
- `sort` — сортирует строки в алфавитном порядке. Это критически важно для `uniq`
- `uniq -u` — выводит только строки без дублей (`-u` = unique)
- `|` — пайп (pipe): передаёт вывод одной команды на вход другой. Фундаментальная концепция Unix

Без предварительной сортировки `uniq` не заметит повторы, разбросанные по файлу.

## Level 10 → 11

**Задача:** `data.txt` содержит данные, закодированные в Base64.

```bash
cat data.txt    # видим: VGhlIHBhc3N3b3JkIGlzIGR...
```

**Объяснение:**
- Base64 кодирует бинарные данные в ASCII строку (A-Z, a-z, 0-9, +, /)

- `-d` — режим декодирования (decode)
- Base64 — не шифрование! Это просто другое представление данных, легко декодируемое

**Объяснение:**

## Level 12 → 13

**Задача:** `data.txt` — hexdump файла, который был сжат несколько раз. Нужно последовательно распаковать.

Самый многошаговый уровень. Нужно терпение.

```bash
mkdir /tmp/mywork && cp data.txt /tmp/mywork/ && cd /tmp/mywork

# Конвертируем hexdump обратно в бинарный файл
xxd -r data.txt > data.bin

# Смотрим тип и распаковываем по кругу
file data.bin
```

Дальше — итеративно:

```bash
# Если gzip:
mv data.bin data.gz && gzip -d data.gz

# Если bzip2:
mv data data.bz2 && bzip2 -d data.bz2

# Если tar:
tar xf data.tar

# Проверяем снова:
file data        # повторяем, пока не получим "ASCII text"
```

**Объяснение:**
- `xxd -r` — конвертирует hexdump обратно в бинарный файл
- `gzip -d` / `bzip2 -d` — распаковка соответствующих форматов
- `tar xf` — распаковка tar (`x` = extract, `f` = file)
- На каждом шаге `file` подсказывает следующий формат

## Level 16 → 17

**Задача:** В диапазоне портов `31000-32000` найти нужный SSL-сервис, отправить пароль и получить приватный SSH ключ.

```bash
# Сканируем порты
nmap -sV localhost -p 31000-32000

# Пробуем SSL порты (ищем тот, что отвечает ключом, а не эхом)
```

- `chmod 400` — права "только чтение для владельца". SSH откажется использовать ключ с более открытыми правами (защита от случайной утечки)

## Level 19 → 20

**Задача:** В домашней директории SUID бинарник `bandit20-do`. Использовать его для чтения пароля.

```bash
ls -la
# -rwsr-x--- bandit20-do (видим 's' в правах)

./bandit20-do cat /etc/bandit_pass/bandit20
```

**Объяснение:**
- SUID (Set User ID) — специальный бит прав. Программа с SUID запускается от имени **владельца файла**, а не от имени запускающего
- `bandit20-do` принадлежит пользователю `bandit20` → при запуске получает его права
- Буква `s` вместо `x` в правах (`rws`) — признак SUID бита
- Поэтому мы можем читать `/etc/bandit_pass/bandit20`, хотя напрямую у нас нет доступа

- `suconnect` подключается, получает пароль, проверяет его и отдаёт следующий
- Можно использовать любой порт выше 1024 (ниже — нужны права root)

## Level 22 → 23

**Задача:** Снова cron, но теперь нужно самим вычислить имя файла с паролем.

```bash
cat /etc/cron.d/cronjob_bandit23
```

- Мы эмулируем, что выполнил бы скрипт от имени `bandit23`

## Level 23 → 24

**Задача:** Написать bash-скрипт, который cron выполнит от имени `bandit24`, и получить пароль.

```bash
mkdir /tmp/b24work
chmod 777 /tmp/b24work

cat > /tmp/b24work/getpass.sh << 'EOF'
```

**Объяснение:**

- `chmod 777 /tmp/b24work` — даём cron'у права на запись в нашу директорию
- Heredoc `<< 'EOF'` — удобный способ записать многострочный текст в файл

## Level 24 → 25

**Задача:** Сервис на порту `30002` принимает пароль `bandit24` + 4-значный PIN (0000–9999). Нужен брутфорс.

```bash
# Генерируем все 10000 комбинаций
for i in $(seq 0 9999); do
    echo "<пароль_bandit24> $(printf '%04d' $i)"
done > /tmp/pins.txt

# Отправляем всё за одно подключение
cat /tmp/pins.txt | nc localhost 30002 | grep -v "Wrong"
```

**Объяснение:**
- `seq 0 9999` — генерирует числа от 0 до 9999
- `printf '%04d'` — форматирует число с ведущими нулями (0001, 0042, 9999...)
- Один большой файл + одно подключение эффективнее, чем 10000 отдельных
- `grep -v "Wrong"` — скрываем строки с "Wrong PIN", оставляем только успешный ответ

- Vim умеет запускать оболочку: `:set shell` устанавливает тип оболочки, `:shell` — открывает её

## Level 31 → 32

**Задача:** Нужно запушить файл `key.txt` с содержимым `May I come in?` в ветку `master`.


## Level 32 → 33

**Задача:** После входа — "UPPERCASE SHELL": всё что вводишь, переводится в верхний регистр. `ls` → `LS` → команда не найдена.

```bash
$0
```

Теперь мы в обычном `sh`:

```bash
cat /etc/bandit_pass/bandit33
```

**Объяснение:**
- `$0` — специальная переменная, содержащая имя текущей выполняемой оболочки/скрипта
- UPPERCASE shell не трогает переменные — `$0` остаётся `$0`, а не `$0` (уже заглавная)
- Когда bash видит `$0`, он запускает текущую оболочку — `sh` — и мы выбираемся
